# 1. ПЕТТЕ ЗАКОНА — НЕПРОМЕНИМИ

## ⚡ ЗАКОН №1 — ПЕШО НЕ ПИШЕ НИЩО

Пешо говори. AI прави. Пешо потвърждава. Клавиатура = **САМО за числа** (цена, количество).

Всеки ред код минава през този филтър. Пешо натиска ЕДИН бутон — AI прави всичко останало. Никакви форми. Никакви въпросници. Voice + tap + photo са единствените входни методи.

Когато Claude започне да пише код и вижда `<input type="text">` за нещо различно от число — **спри. Препиши го през voice/tap/photo.**

### 🔴 ПОД-ЗАКОН №1A — VOICE ВИНАГИ ПОКАЗВА КАКВО Е ЧУЛ

**Voice НИКОГА не изпраща команда без да е показал транскрипцията на Пешо.**

Преди всяко действие:
1. Пешо говори
2. Транскрипцията се показва в реално време (interim results)
3. Пешо вижда точно **какво е чул AI**
4. Пешо потвърждава с "Изпрати →" ИЛИ диктува отново

**Това не е optional feature — това е core trust mechanism.**

Без това Пешо се чувства "съм в ръцете на робота". С това — Пешо е **контролер**, AI е **изпълнител**.

### Правила за voice UX:

1. **Показвай текста докато говори** (interim results = true)
2. **След като спре — показвай финалния текст** с бутон "Изпрати →" и "Диктувай отново"
3. **Ако confidence < 0.8** — показвай текста в жълт цвят, за да знае Пешо че може да има грешка
4. **При низко confidence** — предлагай 2-3 алтернативи (fuzzy match с product names в базата)
5. **НИКОГА** не auto-submit-вай voice команда без визуално потвърждение

### Пример:

```
Пешо казва: "две черни тениски L за 40 лева"
          ↓
Екранът показва (в реално време):
    "две черни..."
    "две черни тениски..."
    "две черни тениски L..."
    "две черни тениски L за 40 лева"
          ↓
След като Пешо спре:
┌─────────────────────────────────┐
│ "две черни тениски L за 40 лева"│
│                                 │
│ [Диктувай отново]  [Изпрати →]  │
└─────────────────────────────────┘
```

Пешо **винаги** вижда какво ще се изпрати **преди** да се изпрати.

## ⚡ ЗАКОН №2 — PHP СМЯТА, AI ГОВОРИ

AI (Gemini) **НИКОГА** не търси в данни. **НИКОГА** не смята. **НИКОГА** не анализира.

- PHP пуска SQL → получава число → подава **ГОТОВ** резултат
- AI САМО облича числото в изречение (на `tenant.language`)
- **Pills** = чист PHP + SQL → мигновено, 0 cost, 0 грешки
- **Signals** = чист PHP + SQL → мигновено, 0 cost, 0 грешки
- **Свободен чат** = AI → САМО когато Пешо пита нещо непокрито от pills/signals

**Съотношение:** 80% тапване на pills/signals, 20% свободен чат.

### 🎯 ТАКСОНОМИЯ НА ЗАЯВКИТЕ (5 класа A-E)

Всяка потребителска заявка попада в **един от 5 класа**. Класа определя дали има AI call или не.

**Class A — Zero-AI (директни данни)**
- "Колко продадох днес?"
- "Има ли Nike 42?"
- "Колко е оборотът?"
- Процес: SQL query + template → отговор
- AI calls: **0**
- Процент от реалните заявки: **~40%**

**Class B — Phrasing Variants (synonym dictionaries)**
- "Колко направих днес?" / "Как върви?" / "Оборотът?" — всички eq на Class A
- Процес: synonym dictionary → canonical intent → SQL + template
- AI calls: **0**
- Процент: **~20%**

**Class C — AI Formatting (числата от PHP, AI само обвива)**
- Числата са сметнати от PHP. AI получава `{sales_today: 847, avg_week: 720}` → пише "Днес: 847 €, 18% над средното."
- Процес: PHP data → AI формулира изречение
- AI calls: **~20%** (но с готови данни — малки prompts)
- Процент: **~20%**

**Class D — Ambiguous / Exploratory**
- "Защо печалбата е по-слаба?"
- "Какво ми тежи най-много?"
- Процес: AI анализ + комбинация на данни
- AI calls: **~100%**
- Процент: **~15%**

**Class E — Open Generation (OCR, marketing)**
- OCR interpretation на фактури
- Marketing текстове за промоции
- Long summaries
- AI calls: **~100%**
- Процент: **~5%**

### 🎯 QUESTION COMPRESSION LAYER

**100 различни изречения → 1 canonical intent.**

```php
// PHP без AI — matches synonyms, dialects, typos
$intent_map = [
    'sales_today' => [
        'колко продадох днес', 'как върви денят', 'оборотът днес',
        'колко взех днес', 'ден', 'какво сме взели днес',
        // + typo tolerance
    ],
    'stock_check' => [
        'имаш ли', 'има ли', 'колко ми остана', 'има ли в наличност',
    ],
    // ... 50-100 canonical intents
];

function compressToIntent($userInput) {
    $normalized = strtolower(trim($userInput));
    foreach ($intent_map as $intent => $phrases) {
        foreach ($phrases as $phrase) {
            // Fuzzy match + typo tolerance
            if (similar_text($normalized, $phrase) > 0.80) {
                return $intent; // → PHP handler, 0 AI cost
            }
        }
    }
    return null; // → fallback to AI
}
```

**Това е ключът към 10% target.** Без Question Compression → всеки различен phrasing = нов AI call.

### 🎯 CONVERSATION STATE MACHINE

**Проблем без това:** Follow-ups мигрират Class A заявки в Class D.

Пример:
- "Колко продадох днес?" → Class A (PHP, 0 AI cost)
- "А вчера?" → **ambiguous** без context → мигрира в Class D → AI call

**Решение:** PHP conversation state в chat session.

```php
class ConversationState {
    public $last_intent;      // 'sales_today'
    public $last_entities;    // ['date' => 'today', 'metric' => 'sales']
    public $last_result;      // [847.50, ...]

    public function resolveFollowUp($userInput) {
        // "А вчера?" → inherit last_intent ('sales_today'), swap date
        if (preg_match('/а (вчера|миналата седмица|този месец)/i', $userInput, $m)) {
            return [
                'intent' => $this->last_intent,
                'entities' => array_merge($this->last_entities, ['date' => $m[1]])
            ];
        }

        // "Само Nike?" → inherit intent, add filter
        if (preg_match('/само ([а-я]+)/i', $userInput, $m)) {
            return [
                'intent' => $this->last_intent,
                'entities' => array_merge($this->last_entities, ['brand' => $m[1]])
            ];
        }

        return null; // Not a follow-up → proceed to AI
    }
}
```

**Без conversation state:** AI calls остават на 25-30%.
**С conversation state:** AI calls падат до 10-15%.

### Механизми за намаляване (общо):

1. **Question Compression Layer** — 100 изречения → 1 intent (без AI)
2. **Conversation State Machine** — follow-ups без AI
3. **biz_learned_data** — AI учи от всеки клиент, споделя с нови (CORE §29)
4. **Pre-computed insights** — cron-15min обновява, AI не се пита (TECH §6)
5. **Template отговори** — за Top 20 въпроса → PHP templates

## ⚡ ЗАКОН №3 — AI МЪЛЧИ, PHP ПРОДЪЛЖАВА

> Ако AI падне → магазинът продължава да работи перфектно.
> Ако числата не съвпадат → AI мълчи.
> Ако контекстът е остарял → AI мълчи.
> Ако терминологията не е чиста → AI мълчи.
>
> **Мълчанието винаги е по-добре от глупостта.**
> Пешо прощава мълчание. Пешо НЕ прощава глупост.

**Презадаване на мисленето:** AI **не е** мозъкът на продукта. AI е **устата** на продукта.

- Мозък = MySQL + PHP + Invariants
- Уста = AI (говори това което мозъкът знае, с човешки език)
- Ако устата започне да измисля → слагаш ѝ намордник (kill switch), не я правиш "по-умна"

### Практически последствия:

- Всеки AI response минава през **Fact Verifier** (сравнява твърденията с DB)
- Ако Verifier не потвърди → AI връща template fallback ("Нямам сигурни данни за това сега.")
- Ако един tenant има 3 поредни провала → AI pauze за 15 минути за него (trust decay)
- Ако глобално (всички tenants) има над 5% провали — **auto kill switch**, AI изключен, PHP продължава

# 2. КОНЦЕПЦИЯТА — КАКВО Е RUNMYSTORE

## 2.1 Позициониране (фундаментално)

**RunMyStore НЕ Е складова програма. НЕ Е касов апарат. НЕ Е онлайн магазин.**

**RunMyStore е служителят, който Пешо никога не е имал** — работи 24/7, помни всичко, брои всичко, предлага какво да направи. Струва €2 на ден.

Въпросът не е "колко струва програмата" — а **"можеш ли без този служител?"**

## 2.2 Проблемът (какво губи Пешо)

Пешо има магазин за дрехи 20 години. Записва в тетрадка. Знае стоката "по усет". Не знае:

- Колко стока има на рафта
- Кое се продава и кое стои
- Колко печели на всеки артикул
- Какво са питали клиентите и го е нямало
- Коя продавачка носи повече и коя раздава маржа
- Че в 3 от 5-те му магазина тази седмица са питали за рокли — а той не продава рокли

**Пешо губи между 500 и 2,000 лв всеки месец** от:
- Стока която свършва без да знае → пропуснати продажби
- Стока която стои и не мърда → замразени пари
- Грешни цени → под себестойност
- Продавачка с големи отстъпки → изтекъл марж
- Търсени артикули които няма → lost demand

**Но не знае, че губи** — защото никога не е виждал числата.

## 2.3 Решението

Пешо инсталира RunMyStore на телефона си. **Не настройва нищо.** Просто продава.

**Ден 1:**
- Натиска микрофона: *"Две черни тениски, 40 лева."*
- AI записва продажбата. Край.
- Вечерта: *"Днес: 287 лв от 6 продажби. Искаш ли да видиш колко са чиста печалба?"*

**Седмица 1:**
- AI знае 34 артикула.
- Показва: *"Дънките ти носят 27 лв чисто на чифт. Тениската — 22 лв."*
- За 20 години Пешо не е знаел.

**Месец 1:**
- AI знае 150+ артикула.
- *"3 артикула свършват тази седмица. Губиш ~440 лв."*
- Вечерен отчет в 21:00 с числа, топ артикули и проблеми.

**Месец 3:**
- AI познава бизнеса: *"Всеки петък продаваш 2× рокли. Утре е петък — провери витрината."*
- Подготвя готови поръчки. Знае кога кой клиент идва.
- Следи коя продавачка колко носи.

**Складът се изгражда сам.** Всяка продажба, всяка доставка, всяко запитване — системата учи. Без въвеждане. Без тетрадки. Без компютър.

## 2.4 Целеви бизнеси

**Нашите (70%+ вероятност за успех):**
- Дрехи женски/мъжки/детски
- Обувки
- Спортни стоки
- Бижута / аксесоари / чанти
- Бельо / чорапи

**Общ знаменател:** артикул на бройка, баркод, размер/цвят, сезонност, доставчик.

**НЕ наши (изключване):**
- Хранителни магазини (твърде много артикули, много малко марж)
- Аптеки (регулации)
- Месарници (бърза развала)
- Електроника (различна бизнес логика, гаранции)

## 2.5 Юридическа позиция (СУПТО)

**RunMyStore НЕ Е СУПТО** защото:
- Не управлява касовия апарат на Пешо
- Не издава фискални бонове
- Не отразява продажби в брой → само движение на наличности
- Пешо удря на касата сам → както е правил 20 години

**НАП официално:** "Не е СУПТО складова програма, която не се намира в обекта и данните се въвеждат впоследствие."

**Ние сме облачна складова програма** — сървър във Франкфурт, данните се въвеждат по-късно.

### Техническо решение за България:

В базата данни → таблица `stock_movements` (движение на наличности), не `sales`. В български интерфейс → терминология "изходящо движение", "намаление на количество". В целия останал свят → нормална "продажба" с цена и детайли.

Един флаг в кода: `if ($tenant['country_code'] == 'BG')` → различна терминология в UI и справки.

### Правна защита:

Общи условия, параграф: *"Потребителят е отговорен за всички данъчни задължения. Нашият софтуер не замества касовия апарат."*

**Консултация с данъчен адвокат** преди старт в България → ~200-300 лв (еднократно).

**СУПТО сертификация — цел 2027-2028** за по-големи клиенти които я искат. Не е приоритет в момента.

### loadTranslations() helper:

```php
function loadTranslations($lang) {
    static $cache = [];
    if (isset($cache[$lang])) return $cache[$lang];

    $path = BASE_PATH . "/lang/{$lang}.json";
    if (!file_exists($path)) {
        $path = BASE_PATH . "/lang/en.json"; // fallback
    }

    $data = json_decode(file_get_contents($path), true);

    // Flatten nested keys: "products.add_new" → value
    $flat = [];
    array_walk_recursive($data, function($v, $k) use (&$flat, $data) {
        $flat[$k] = $v;
    });

    // По-добро: dot-notation lookup
    $cache[$lang] = $data;
    return $data;
}

function t($key, $tenant, $params = []) {
    $lang = $tenant['language'] ?? 'bg';
    $translations = loadTranslations($lang);

    // Dot notation: "products.add_new"
    $value = $translations;
    foreach (explode('.', $key) as $part) {
        if (!isset($value[$part])) {
            // Fallback to English
            if ($lang !== 'en') {
                return t($key, ['language' => 'en'] + $tenant, $params);
            }
            return $key; // Last resort: return the key itself
        }
        $value = $value[$part];
    }

    foreach ($params as $k => $v) {
        $value = str_replace("{{$k}}", $v, $value);
    }
    return $value;
}
```

## 3.2 Правило EUR (валута в България)

**България е в евро от 1.1.2026** (вече!). Пешо вече работи в евро.

### Важни дати:

| Дата | Статус |
|---|---|
| 1.1.2026 | BG официално в eurozone. Основна валута = EUR. |
| 1.1.2026 → 8.8.2026 | **Двойно обозначаване** (€ + лв) **задължително по закон** — фиксиран курс **1.95583** |
| **8.8.2026** нататък | Само € (без лв) |

### Железни правила:

1. **НИКОГА** hardcoded "лв", "€", "$" в PHP, HTML, JS
2. Винаги през функция `priceFormat($amount, $tenant)`
3. Функцията гледа: `tenant.currency` + `tenant.country_code` + **днешната дата**
4. Fallback currency = **EUR** (не BGN вече!)
5. Стари данни с `currency='BGN'` → AI конвертира при показване (÷ 1.95583)

### Имплементация:

```php
function priceFormat($amount, $tenant) {
    $currency = $tenant['currency'] ?? 'EUR';
    $country = $tenant['country_code'] ?? 'BG';
    $lang = $tenant['language'] ?? 'bg';
    $today = date('Y-m-d');

    // Специален случай: България между 1.1.2026 и 8.8.2026
    if ($country === 'BG' && $today < '2026-08-08') {
        $eur = number_format($amount, 2, ',', '.');
        $bgn = number_format($amount * 1.95583, 2, ',', '.');
        return "{$eur} € / {$bgn} лв";
    }

    // Нормално форматиране
    $symbols = ['EUR' => '€', 'USD' => '$', 'GBP' => '£', 'RON' => 'lei', 'PLN' => 'zł'];
    $symbol = $symbols[$currency] ?? $currency;

    // EU формат: 1.234,50
    $formatted = number_format($amount, 2, ',', '.');
    return "{$formatted} {$symbol}";
}

// Употреба:
echo priceFormat(29.99, $tenant);
// BG преди 8.8.2026: "29,99 € / 58,65 лв"
// BG след 8.8.2026: "29,99 €"
// DE: "29,99 €"
// US: "29,99 $"
```

### На етикети за печат:

Same rule — двойно обозначаване € + лв до 8.8.2026, после само €. Wizard step 7 има radio бутон *"€ + лв / Само € / Без цена"* — защото:
- Някои продавачи предпочитат само евро вече
- Някои искат двойно за клиентите
- Някои нямат цени на етикета

**Default** (България, преди 8.8.2026): **€ + лв** (законово изискване).

### Проверка за hardcoded валути:

Преди всеки commit: `grep -rn "лв\|BGN" *.php` → трябва да върне нула резултати извън priceFormat() функцията.

## 3.3 Как мислим за новите функции

Всяка нова функция минава проверка:

1. **Работи ли в 20 езика?** Ако има hardcoded текст → спри.
2. **Работи ли с EUR, USD, RON, PLN?** Ако има hardcoded валута → спри.
3. **Работи ли на `country=DE`?** Ако има BG-специфична логика (като СУПТО терминология) → увий я в `if ($tenant['country_code'] === 'BG')`.
4. **Работи ли в timezone `UTC+2` и `UTC-5`?** Ако има hardcoded `Europe/Sofia` → ползвай `$tenant['timezone']`.

## 6.10 ЛЕГАЛНА СТРУКТУРА — INDEPENDENT SOFTWARE RESELLER (ISR)

### Правната реалност

В EU (Директива 2005/29/EC) **MLM/pyramid** се дефинира като:

> *"Consumer gives consideration (plащане) for the chance to receive compensation derived primarily from the introduction of OTHER consumers, rather than from sale/consumption of products/services."*

**RunMyStore model НЕ попада под това определение** защото:

1. ✅ **Comissионите идват от реални subscription sales** — не от recruitment event
2. ✅ **Няма входни такси** — партньорите не плащат да станат партньори
3. ✅ **Няма inventory purchases** — не купуват нищо от нас
4. ✅ **Максимум 2 нива** — Territory + Referral, никога 3-4+
5. ✅ **Time-limited referral bonus** — 50% × 6 месеца е CPA модел, не perpetual pyramid income

### Репакетирана структура — ISR модел

**Територeн партньор = Independent Software Reseller (ISR):**

```
RunMyStore (Platform SaaS)
    │
    │ Software License Agreement + Service Agreement
    ↓
Territory Partner (ISR — Independent Software Reseller)
    │
    │ Правата му включват:
    │ ├─ Право да продава софтуера в ексклузивна територия
    │ ├─ 15% revenue share от customers в територията му
    │ ├─ Локална localization и support authority
    │ └─ Достъп до sales materials и training
    │
    │ Service obligations:
    │ ├─ First-line support (< 24h response)
    │ ├─ Локална localization (translations, formats)
    │ ├─ Monthly activity minimum (10 нови customers)
    │ └─ Quality monitoring (NPS > 50 за customers)
```

### Две независими програми (НЕ hierarchy!)

**КРИТИЧНА ПРОМЯНА:** Вместо "Territory партньорът управлява Sub-партньорите" → **две независими програми**.

| Program 1: Territory License | Program 2: Referral Bounty |
|---|---|
| Ексклузивна регионална license | **Отворено за всеки** (POS сервизи, счетоводители, и т.н.) |
| 15% от ВСИЧКИ customers в територията | 50% × 6 месеца за **директен referral** |
| Изисква service obligations | Няма service obligations |
| Service Agreement | Referral Agreement |
| Дългосрочен (years) | Per-customer (еднократна сделка) |
| За business owners | За individuals (може и бизнес) |

**Territory Partner И Referral Partner могат да са един и същ човек** — но получават commissions от **две различни програми**, а не от "йерархия".

### Забранен MLM vocabulary

**❌ Никога не се ползват думи:**
- "downline", "upline"
- "network", "team tree"
- "levels", "generation"
- "recruiting commission"

**✅ Разрешен vocabulary:**
- "reseller", "territory partner"
- "referral partner", "channel partner"
- "commission", "revenue share"
- "territory license", "territory rights"

### Audit trail — documentation requirement

Всеки payout трябва да има:

```sql
CREATE TABLE commission_audit (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    partner_id INT NOT NULL,
    payment_date DATE,
    amount_cents INT,
    partner_type ENUM('territory', 'referral') NOT NULL,

    -- АНТИ-MLM justification
    customer_id INT NOT NULL,           -- конкретен клиент
    subscription_id INT NOT NULL,       -- конкретен договор
    invoice_id VARCHAR(100),            -- конкретна фактура
    service_period_start DATE,
    service_period_end DATE,
    service_rendered TEXT,              -- "territory support", "referred customer acquisition"

    INDEX (partner_id, payment_date)
);
```

Ако някога бъдем запитани от Stripe/регулатор → **за всяка €** имаме конкретно **"този клиент, за този период, за тази услуга"**.

### Защита пред Stripe

В Stripe Connect metadata винаги описваме:

```php
'partner_type' => 'reseller',           // НЕ 'affiliate'
'service_type' => 'software_resale',    // НЕ 'recruitment_bonus'
'contract_type' => 'isr_agreement'      // Independent Software Reseller
```

Stripe Express акаунтите ни гледат като **SaaS reseller platform**, не като MLM.

### Защита в медии/PR

Ако някой публично нарече нашия модел MLM → **отговорът е:**

> *"Referral partners получават 50% commission за 6 месеца — това е стандартен CPA (Cost Per Acquisition) модел, използван от Shopify, HubSpot, Dropbox, и всички SaaS индустрии. Не е perpetual passive income.*
>
> *Territory partners получават 15% от customers в тяхната територия, защото извършват реални услуги (localization, first-line support, quality monitoring) — точно като McDonald's franchise или Insurance broker network.*
>
> *Никой не плаща такси да стане партньор. Никой не купува inventory. Commissions идват само от реални subscription sales на software."*

### Защо това работи

Това е аналогично на:

- **McDonald's franchise** — master franchisee + sub-franchisees, всички получават дял
- **Coca-Cola bottling network** — regional distributors + local sellers
- **Insurance broker hierarchies** — lead broker + sub-brokers
- **Real estate agent networks** — broker + agents

**Юридическа форма:** ISR Agreement + Service Agreement — НЕ employment, НЕ MLM, НЕ direct sales compensation plan.

# 18. AI NAVIGATOR — AI КАТО СИСТЕМЕН ВОДАЧ

## 18.1 Концепция (от S71)

AI познава **ВСЕКИ модул, екран, бутон**. В "Опростен" режим **AI Е менюто**.

### Пример:
Пешо пита: *"Как добавям нов артикул?"*

❌ Стар подход:
*"Натисни 3-те точки → Артикули → + Добави → Wizard → Стъпка 1..."*

✅ Нов подход (AI Navigator):
```
"Добавяш с глас: 'Бяла блуза, 29 лева, 5 броя'
или натискаш [Добави артикул ↓] — аз ще те водя."

[Добави артикул ↓]   ← Action бутон директно
```

## 18.2 Action бутони навсякъде

Всеки AI отговор може да завърши с **Action бутон** който води директно:

```
"3 артикула на нула. Nike 42, Adidas 38, Puma 41.
Обмисли поръчка."

[Виж артикулите →]   [Подготви поръчка →]
```

### Техническа реализация:

**build-prompt.php Layer 9:**
```php
$prompt .= "\n\nIMPORTANT: Finish your response with appropriate action buttons.
Format: [[ACTION: url_or_intent | label]]
Examples:
  [[ACTION: products.php?filter=zero_stock | Виж артикулите]]
  [[ACTION: order_draft:nike_42,adidas_38 | Подготви поръчка]]
  [[ACTION: navigate:deliveries | Отвори Доставки]]
";
```

**chat-send.php:** Хваща `[[ACTION: ...]]` → парсва → генерира HTML бутони.

## 18.3 Deeplinks от чата

Всеки action бутон е **deeplink към конкретен екран/състояние**:

| Action | Води до |
|---|---|
| `products.php?filter=zero_stock` | Списък артикули с наличност 0 |
| `products.php?filter=zombie` | Zombie стока |
| `products.php?filter=below_cost` | Под себестойност |
| `sale.php?product_id=X` | Касата с предизбран артикул |
| `deliveries.php?supplier_id=X` | Нова доставка от доставчик X |
| `order_draft:nike_42,adidas_38` | Чернова поръчка с артикулите |
| `inventory.php?zone_id=X` | Zone Walk на конкретна зона |

## 18.4 Философия за контрола

**AI ВОДИ, но Пешо ВИНАГИ може да каже "не, искам ръчно".**

Навсякъде има **"Ръчен режим →"** бутон — escape hatch за опитни потребители.

Философията от онбординга: *"Водим, не показваме"* — но НЕ *"Водим принудително"*.

## 20.1 Водим, не показваме (но не принудително)

Повечето SaaS продукти хвърлят потребителя в dashboard с 12 бутона и очакват да се ориентира. RunMyStore прави обратното:

**"Водим Пешо стъпка по стъпка — но винаги може да каже 'искам сам'."**

### Ключови принципи:

1. **Договор с Пешо** (не туториал) — визуален timeline с обещания какво получава кога
2. **Минимален input в signup** — само email (за retargeting + support)
3. **Физическа първа победа** — Bluetooth етикет в ръката, не скрийншот
4. **Прогресивно разкриване** — всички модули достъпни от ден 1 чрез "Всички инструменти" drawer, но главният екран фокусира върху следващия milestone
5. **Escape hatch навсякъде** — "Ръчен режим" бутон винаги видим

## 20.2 Два системни режима

DB колона:
```sql
tenants.onboarding_status ENUM('new','in_progress','core_unlocked','operating') DEFAULT 'new'
tenants.onboarding_milestones JSON  -- {first_product: "2026-04-20", first_label: "2026-04-20", ...}
```

| Режим | Главен екран | Кога се активира |
|---|---|---|
| **new** | Welcome hook → Signup | Ден 0 |
| **in_progress** | 3 врати + milestones чеклист | След signup, до завършване на milestones |
| **core_unlocked** | 3 врати (голяма OCR) + pulse signals | След първа продажба + 48ч |
| **operating** | Full dashboard (chat.php или simple.php) | След 30+ дни активност |

### Helper функция:

```php
function getUserMode($tenant) {
    $status = $tenant['onboarding_status'] ?? 'new';
    if ($status === 'operating') return 'operating';

    $milestones = json_decode($tenant['onboarding_milestones'] ?? '{}', true);
    $has_products = !empty($milestones['products_added']) && $milestones['products_added'] >= 3;
    $has_label = !empty($milestones['first_label']);
    $has_sale = !empty($milestones['first_sale']);
    $days_active = floor((time() - strtotime($tenant['created_at'])) / 86400);

    if ($has_products && $has_label && $has_sale && $days_active >= 2) {
        // Milestones завършени → core_unlocked
        return 'core_unlocked';
    }
    if ($days_active >= 30) return 'operating';
    return 'in_progress';
}
```

# 25. STORE HEALTH = CTA МОТОР

## 25.1 Какво е Store Health

**НЕ е "% инвентаризация" — това е досадно.**
**Е "% колко AI знае за магазина ти" — лично и конкретно.**

### Формула:

```php
function storeHealth($tenant_id, $store_id): int {
    // Stock accuracy (40%): % артикули проверени в последните 30 дни
    $total = DB::run("SELECT COUNT(*) FROM products
                      WHERE tenant_id=? AND is_active=1", [$tenant_id])->fetchColumn();
    $confirmed = DB::run("SELECT COUNT(*) FROM products
                          WHERE tenant_id=? AND is_active=1
                          AND last_counted_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)",
                          [$tenant_id])->fetchColumn();
    $accuracy = $total > 0 ? ($confirmed / $total) * 100 : 0;

    // Data freshness (30%): обратно на дни от последен zone walk
    $avg_days = DB::run("SELECT AVG(DATEDIFF(NOW(), COALESCE(last_walked_at, created_at)))
                         FROM store_zones
                         WHERE tenant_id=? AND store_id=?",
                         [$tenant_id, $store_id])->fetchColumn() ?: 30;
    $freshness = max(0, 100 - ($avg_days * 3)); // -3% на ден

    // AI confidence (30%): средна confidence_score
    $avg_conf = DB::run("SELECT AVG(confidence_score) FROM products
                         WHERE tenant_id=? AND is_active=1",
                         [$tenant_id])->fetchColumn() ?: 0;

    return round(($accuracy * 0.4) + ($freshness * 0.3) + ($avg_conf * 0.3));
}
```

# 26. LOST DEMAND — ЦЕНТРАЛЕН DIFFERENTIATOR

## 26.1 Концепция

> **"POS системите знаят какво СИ ПРОДАЛ.
> Никоя не знае какво НЕ СИ ПРОДАЛ.
> Ние знаем."**

Това е **уникален** за пазара feature. Никой друг POS/склад софтуер не го прави.

## 26.2 Как се събират данни

### Автоматично (Петя не прави нищо):

```
1. Клиент пита за "бели маратонки 38"
2. Петя отваря sale.php, търси "бели маратонки 38"
3. search_log записва: query="бели маратонки 38", results_count=0
4. Автоматично → lost_demand: query_text="бели маратонки 38", source="search"
5. Петя казва "нямаме" — не натиска нищо допълнително
```

### Ръчно (бутон "Какво търсиха?"):

```
1. В chat.php: бутон/chip "Какво търсиха?"
2. Петя натиска → voice overlay → казва "Търсиха рокли за кръщене и детски ботуши 32"
3. AI парсва свободния текст → записва в lost_demand:
   - query_text="рокли за кръщене", source="voice"
   - query_text="детски ботуши 32", source="voice"
4. НЕ търси match в products! Записва СВОБОДЕН ТЕКСТ.
```

### Защо свободен текст:

Клиентът може да търси нещо което **изобщо не е в каталога**. "Рокли за кръщене" — Пешо никога не е продавал рокли. Ако търсим match → нищо не намираме → **данните се губят**. Свободният текст запазва ТОЧНО какво е казал клиентът.

## 26.3 Как се показват данни

### Седмичен pill на home:
```
🟡 "5× търсиха бели маратонки 38"
```

### Cross-store signal (БИЗНЕС план):
```
🔴 НОВА КАТЕГОРИЯ СИГНАЛ
В 3 от 5 магазина тази седмица търсиха РОКЛИ.
9 клиента за 2 седмици. Никога не си продавал рокли.
Помисли.
```

### При поръчка:
```
🟡 Подготвяш поръчка за Иватекс.
Ето какво търсиха клиенти тази седмица:
- Бели маратонки 38 (5 пъти, 2 магазина)
- Черни дънки 34 (3 пъти)
- Детски ботуши 32 (2 пъти)
Помисли да ги добавиш.
```

## 26.4 DB schema

```sql
CREATE TABLE search_log (
    id INT AUTO_INCREMENT PRIMARY KEY,
    tenant_id INT NOT NULL,
    store_id INT NOT NULL,
    user_id INT NOT NULL,
    query VARCHAR(255) NOT NULL,
    results_count INT DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_zero (tenant_id, results_count, created_at)
);

CREATE TABLE lost_demand (
    id INT AUTO_INCREMENT PRIMARY KEY,
    tenant_id INT NOT NULL,
    store_id INT NOT NULL,
    user_id INT DEFAULT NULL,
    query_text VARCHAR(500) NOT NULL,
    source ENUM('search','voice','manual') DEFAULT 'search',
    matched_product_id INT DEFAULT NULL,   -- попълва се при поръчка на подобен артикул
    resolved TINYINT(1) DEFAULT 0,
    first_asked_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_asked_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    times INT DEFAULT 1,
    INDEX idx_tenant_resolved (tenant_id, resolved, created_at)
);
```

## 26.5 Мотивация на Петя

**Петя не работи за Пешо. Петя работи за себе си.**

- Петя записва "Търсиха рокли"
- Пешо поръчва рокли
- Стоката идва
- Петя ги продава
- Петя взима процент от продажбата

AI казва на Пешо:
```
"Петя записа 12 търсения този месец.
От тях поръча 4, продаде 3 за 180 €.
Тези пари дойдоха заради Петя.
Мария не е записала нищо."
```

## 26.6 ROI tracking (lost demand → sale)

```sql
-- Lost demand водещ до поръчка и продажба
SELECT ld.query_text, p.name,
       SUM(si.quantity) as sold,
       SUM(si.unit_price * si.quantity) as revenue
FROM lost_demand ld
JOIN products p ON p.id = ld.matched_product_id
JOIN sale_items si ON si.product_id = p.id
JOIN sales s ON s.id = si.sale_id AND s.created_at > ld.first_asked_at
WHERE ld.tenant_id=? AND ld.matched_product_id IS NOT NULL
GROUP BY ld.query_text, p.id;
```

Това дава:
*"Записано търсене 'бели маратонки 38' → поръчани 10 бр → продадени 7 за 280 €. ROI: 280 € от 1 tap на Петя."*
