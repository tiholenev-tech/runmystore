## ЦЕЛ

Изграждане на нощна симулационна система която всяка вечер живее един работен ден на фирма с пет обекта плюс онлайн, единадесет доставчика, петима продавачи. Сутрин дава отчет на Тихол какво работи, какво гърми, къде сметката не излиза.

**Защо изобщо строим тази система:** Тихол открива магазин с голям обем стока. Инвентаризацията трябва да работи 1000% сигурно от ден едно. Ако нещо се обърка в началото и се бъгне, после трябва да се връщат и да броят всичко наново — катастрофа. Стрес тестът е защитата срещу този сценарий. Без него Тихол не пуска beta.

Стрес системата работи паралелно с основния проект. Тества старото построено и новото построено през деня едновременно.

## ЧЕТИРИ ЖЕЛЕЗНИ ЗАКОНА

### ЗАКОН 1 — ЧЕТЕНЕ НА ДЪСКАТА

Никой чат не започва работа без да прочете STRESS_BOARD.md първо. Без изключение. Без този закон кръгът не се затваря — дневният чат пропуска бъга, Стрес чатът пропуска оправеното, Шеф чатът пропуска отчета.

### ЗАКОН 2 — CHANGES LOG

След всяка Code сесия, преди handoff да се счита за завършен, се пише задължителен файл CHANGES_LOG_sXX.md с три графи.

Първа графа: какво трябваше да направя (едно изречение от задачата).

Втора графа: какво направих точно (списък с всички пипнати файлове, всеки с едно изречение какво е променено, без изключения, включително "създадох таблица", "изтрих доставчик", "смених цвят").

Трета графа: какво НЕ направих въпреки че можеше да изглежда нужно.

Допълнително: главен файл CHANGES_LOG.md, един ред на сесия, хронология в един екран. Шеф чатът го чете преди да дава нова задача.

Причина за закона: Code чат веднъж промени мълчаливо доставчици, категории и под-категории. Открито десет дни по-късно.

### ЗАКОН 3 — ПРОТОКОЛ "ПИСАНЕ НА ДОКУМЕНТ"

При стартиране на нов чат, първото нещо което чатът пише е: "Виждам Протокол Писане на Документ. Ще го изпълнявам само когато Тихол го извика преди документ. Без негова команда — пиша нормално."

Командата на Тихол: "изпълни протокол писане на документ".

Седем стъпки:

Стъпка 1 — прочитам целия разговор отгоре до долу, бавно, без скачане.

Стъпка 2 — изваждам списък с всички решения, факти, договорки, бъгове, имена, числа.

Стъпка 3 — прочитам втори път, допълвам списъка с пропуснатото.

Стъпка 4 — прочитам трети път, търся противоречия между ранно и късно казано, записвам кое е финалното решение.

Стъпка 5 — чак сега пиша документа.

Стъпка 6 — преди изпращане сравнявам с списъка, всяка точка покрита ли е.

Стъпка 7 — в края задължително пиша един от два реда: "ПРОТОКОЛ ИЗПЪЛНЕН: Минах разговора три пъти. Списъкът е покрит изцяло." или "ПРОТОКОЛ НЕ Е ИЗПЪЛНЕН ДОКРАЙ: Минах [един/два] пъти. Пропуснах [причина]. Възможно е да липсват точки."

Без увъртане. Без украса. Самопризнание: ако Тихол попита "изпълни ли протокола?" отговорът е честен винаги.

### ЗАКОН 4 — TENANT 99 RESET (cron 0 в 01:00)

Симулираните данни в tenant 99 се изтриват по график:

DELETE FROM sales, sale_items, deliveries, transfers, inventory_movements WHERE tenant_id=99 AND created_at < NOW() - INTERVAL 60 DAYS;

Запази 60-дневен прозорец активни данни (за зомби сигнали и сезонни патърни).

Не пипай master данни — products, suppliers, stores, users в tenant 99. Те се възстановяват само от Етап 1 (когато се пресъздаде света).

Без този cron tenant 99 пухне до 5-10GB за месец и мъчи MySQL заявките.

## CODE НЕ ОПРАВЯ САМ

Code чат прави големи бели мълчаливо. Затова разрешените и забранените действия са фиксирани.

Code може да оправя сам: грешен текст, грешен превод, липсваща null проверка, забравен параметър, грешен ключ в SQL заявка. Тривиални неща в един и същи файл.

Code НЕ оправя сам: race conditions, схема на базата, дизайн, AI prompts, Simple Mode, нова бизнес логика. Нищо което се отразява на Пешо.

Сутрин в дъската се виждат три категории: оправени автоматично с списък, чакат твоето решение средни, спешни за човек големи.

## СТРУКТУРА НА ЧАТОВЕТЕ

Пет паралелни чата с фиксирани роли.

Чат 1 — Шеф чат. Център на всичко. Координира. Не пише код. Сутрин чете отчета, разпределя задачи.

Чат 2 — Code за AI Brain. Само AI логика, prompts, insights, AI Studio.

Чат 3 — Code за модули и бъгове. sale, products, deliveries, transfers, orders, inventory.

Чат 4 — Opus 4.7. Нови модули и дизайн.

Чат 5 — Стрес чат. Държи документацията на стрес теста. Вечер инструктира cron-овете. Сутрин чете резултатите.

Тихол отваря чатове, прави copy-paste, мисли стратегията. Не пише сам нищо.

**"Дневен чат" в STRESS_BOARD граф 1** = всеки чат който е завършил задача през деня (Code за AI Brain, Code за модули, Opus). НЕ Шеф чатът — той координира, не строи. Който е оправил бъг или построил модул → той пише в граф 1 "тествай тая вечер".

## ДНЕВЕН РЕД

Сутрин — Шеф чатът чете нощния отчет, разпределя задачи на Code чатовете и Opus.

През деня — строят се нови неща.

Вечер преди затваряне — Тихол отваря Стрес чата, той чете Компаса, пише инструкции на cron-овете, затваря деня.

Нощем — cron-овете работят сами без човек.

Сутрин пак — Шеф чатът чете отчета и кръгът започва отначало.

## ТРИ CRON-А — И ЗАЩО ТРИ, А НЕ ЕДИН

Три отделни файла, три различни часа, един флаг между тях — следващият не започва ако предишният не е готов.

Cron 1 в 02:00 нощта — симулира деня в петте обекта, стотици продажби, доставки, трансфери, инвентаризации, хиляди записи в базата.

Cron 2 в 03:00 — тества модулите от графата "за тестване", атакува новата функционалност построена днес.

Cron 3 в 06:00 — прави сутрешния отчет от това което двамата преди него са направили.

Бонус — Claude Code в 06:30, чете отчета, анализира с пълен контекст от документацията, пише в дъската не "имаш бъг", а "бъг в файл X ред Y, свързан с параграф Z от Inventory v4".

**Защо три отделни cron-а, а не един голям:**

Първо — изолация при срив. Ако симулацията гръмне в 02:30, отчетът в 06:00 пак се прави с каквото има. Ако всичко беше в един cron — една грешка убива целия процес и сутрин Тихол отваря празен отчет.

Второ — различна логика, различен живот. Симулацията пише в базата. Тестът на новото чете и атакува. Отчетът само чете и обобщава. Разделени отговорности значи разделени файлове.

Трето — лесна поддръжка. Ако трябва да промениш отчета, не пипаш симулацията. Ако симулацията расте, не пипаш отчета.

Четвърто — възможност един от тях да работи вечер ръчно. Ако Тихол иска бърз тест извън графика, пуска само Cron 2 без да чака другите.

**Това е дизайн решение, не препоръка. Не се обединяват в един cron.**
