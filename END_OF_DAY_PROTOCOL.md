# 🌙 ПРОТОКОЛ ЗА КРАЯ НА СЕСИЯТА

**Версия:** 2.1 (02.05.2026)  
**Заменя:** v1.0, v2.0  
**Принцип:** Шефът води Тихол стъпка по стъпка. БЕЗ прескачане. Тихол не знае git — шефът дава точните команди copy-paste.

---

## 🎯 КАК СЕ СТАРТИРА

Тихол пише една от тези фрази на шеф-чата:

- "край на сесията"
- "приключи"

Шефът отговаря със СТЪПКА 1 и спира да чака.

---

## 📋 СТЪПКА 1 — ШЕФЪТ ПРАВИ ДОКУМЕНТИТЕ

Шефът:
1. Минава chat-а 3 пъти (Document Protocol)
2. Генерира 2 файла:
   - `PRIORITY_TODAY.md` (план за утре)
   - `MASTER_COMPASS_UPDATE.md` (logic log + rework queue entries)
3. Дава ги като download links
4. Пише на Тихол ТОЧНО ТОВА:

> **СТЪПКА 1 ГОТОВА.** Свали двата файла на компютъра си.  
> Когато са свалени → пиши "свалих".

**Шефът чака. Не пише нищо повече.**

---

## 📋 СТЪПКА 2 — TIHOL КАЧВА ПРЕЗ FILEZILLA

Когато Тихол пише "свалих", шефът пише ТОЧНО ТОВА:

> **СТЪПКА 2.** Отвори FileZilla. Свържи се с droplet.  
> Drag-ни двата файла от своя компютър в `/var/www/runmystore/` на droplet-а.  
> FileZilla може да пита "Презапис?" — натисни **Презапис** (OK).  
> Когато двата файла са качени → пиши "качих".

**Шефът чака.**

---

## 📋 СТЪПКА 3 — ШЕФЪТ ПРОВЕРЯВА ЧЕ ФАЙЛОВЕТЕ СА НА ДРОПЛЕТА

Когато Тихол пише "качих", шефът пише ТОЧНО ТОВА:

> **СТЪПКА 3.** Отвори droplet web console (DigitalOcean cloud).  
> Copy-paste тази команда (целия ред) и натисни Enter:
> ```
> cd /var/www/runmystore && ls -lh PRIORITY_TODAY.md MASTER_COMPASS_UPDATE.md
> ```
> Покажи ми output-а (текста който излиза в терминала).

Шефът проверява в output-а:
- Двата файла съществуват
- Размерите им изглеждат разумни (PRIORITY ~10-25KB, COMPASS_UPDATE ~5-20KB)
- Дата на последна модификация е от тази сесия

Ако всичко OK → СТЪПКА 4.  
Ако нещо не е наред → шефът казва точно какво не е наред.

---

## 📋 СТЪПКА 4 — TIHOL ПУСКА GIT COMMIT + PUSH

**ВАЖНО за шефа:** Тихол не знае git. Не казвай "пусни git push" — дай му точната команда копи-пейст.

Шефът пише ТОЧНО ТОВА:

> **СТЪПКА 4.** В същия терминал copy-paste тази команда (целия ред) и натисни Enter:
> ```
> cd /var/www/runmystore && git add PRIORITY_TODAY.md MASTER_COMPASS_UPDATE.md && git commit -m "EOD update DD.MM" && git push origin main
> ```
> (Замени `DD.MM` с днешна дата, например `01.05`)
> 
> Тази команда прави 3 неща наведнъж:
> 1. Добавя файловете в git
> 2. Записва ги (commit)
> 3. Праща ги на GitHub (push)
> 
> Покажи ми output-а.

**Шефът чака.**

**Възможни проблеми и решения:**

- **Ако пише "Username for github":** push не върви защото няма auth. Шефът казва: "Натисни Ctrl+C. Пусни команда `exit` за да излезеш на root. После пусни същата команда отново." (Root user има auth-а.)

- **Ако пише "Author identity unknown":** липсва git user config. Шефът дава една команда:
  ```
  git config --global user.name "tiholenev-tech" && git config --global user.email "tiholenev@gmail.com"
  ```
  После казва: "Сега пусни същата команда от стъпка 4 пак."

- **Ако пише "nothing to commit":** файловете не са били променени (или вече са committed). Шефът проверява дали наистина са качени правилно през FileZilla.

## 📋 СТЪПКА 5 — ШЕФЪТ ВЕРИФИЦИРА GITHUB

Шефът използва инструменти да провери GitHub:
- Последен commit hash
- Двата файла са там
- Размерите им са правилни
- Заглавията са правилни (PRIORITY за УТРЕШНА дата, COMPASS_UPDATE за ДНЕШНА)

Ако всичко OK → СТЪПКА 6 (финал).  
Ако нещо не е наред → шефът казва точно какво и дава нова команда.

## ❌ КАКВО ШЕФЪТ НЕ ПРАВИ

- НЕ прескача стъпки. Една стъпка наведнъж.
- НЕ пита Тихол къде е качил → шефът проверява сам с команди
- НЕ казва "качено" преди да види реалния git log
- НЕ дава 3 опции за избор → дава една ясна команда
- НЕ обсъжда дали да включи нещо → решава сам
- НЕ казва "пусни git push" без да даде точната команда
- НЕ предполага че Тихол знае какво е commit/push/branch

## 🚨 EDGE CASES

### А) Тихол излиза рано без "край на сесията"
Шефът сам затваря в края на context-а. Пише само STATE_OF_DAY_[DATE].md. Не претендира за пълен EOD.

### Б) Малко работа през деня
Skip MASTER_COMPASS_UPDATE.md ако няма реални logic changes. Само PRIORITY_TODAY.md (СТЪПКА 1 → 1 файл вместо 2). Командата в СТЪПКА 4 става съответно с само един файл.

### В) Тихол спира насред протокола
Шефът чака без да повтаря. Когато Тихол се върне → продължава от стъпката където са спрели.

### Г) Команда гърми / output не е OK
Шефът диагностицира конкретното несъответствие. Дава една нова команда (точна, copy-paste). Не започва наново.

### Д) Code Code още работи в края
Шефът завършва EOD без да чака. Във PRIORITY_TODAY.md отбелязва: "[N] активни Code Code сесии при затваряне — verify утре сутрин."

## ✅ ПРОВЕРКА ЧЕ ПРОТОКОЛЪТ РАБОТИ

В края на правилен EOD трябва:
1. Тихол не е писал нищо освен "край на сесията", "свалих", "качих", и output-и от 2 команди
2. Има 1 commit на GitHub с двата файла
3. Шефът е казал "Благодаря ти че бяхме заедно"
4. Никой не е питал "къде си качил"

Ако някоя от тези не е изпълнена → протоколът е нарушен. Тихол може да каже "счупи протокола" → шефът признава грешката и поправя.
